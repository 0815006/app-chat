## 1. 环境要求

| 工具 | 版本要求 | 用途 |
|------|----------|------|
| Node.js | >= 18 | Vue 3 + Vite 前端 |
| npm | >= 9 | 包管理 |
| Docker Desktop | >= 24 | 运行本地 Supabase 容器 |
| Supabase CLI | >= 1.100 | 管理 Supabase 实例 |

### 1.1 安装 Supabase CLI

```bash
npm install -g supabase
```

验证安装：

```bash
supabase --version
```

## 2. 首次启动（完整流程）

### 2.1 克隆项目 & 安装前端依赖

```bash
cd d:\ProjectDir\app-chat\client-chat-tauri
npm install
```

### 2.2 启动本地 Supabase

```bash
cd d:\ProjectDir\app-chat\supabase
supabase start
```

首次启动会自动拉取 Docker 镜像（约 2~5 分钟），等待输出：

```
Started supabase local development setup.
```

启动成功后，终端会显示关键连接信息：

| 服务 | 本地地址 | 说明 |
|------|----------|------|
| Studio (管理面板) | `http://127.0.0.1:54323` | 可视化管理 DB、Auth、Storage |
| Kong API 网关 | `http://127.0.0.1:54321` | **前端 SDK 连接地址** |
| PostgreSQL | `127.0.0.1:54322` | 数据库直连 |
| Publishable Key | `sb_publishable_...` | **写入 `.env.development`** |

### 2.3 配置环境变量

确认 `client-chat-tauri/.env.development` 中的值正确：

```env
VITE_SUPABASE_URL=http://127.0.0.1:54321
VITE_SUPABASE_ANON_KEY=<启动日志中的 Publishable Key>
VITE_BACKEND_TYPE=supabase
VITE_GO_API_BASE=http://127.0.0.1:8080/api
VITE_GO_WS_URL=ws://127.0.0.1:8080/ws
```

> **注意**：`VITE_SUPABASE_URL` 是 Kong API 网关端口（默认 54321），不是 PostgreSQL 端口（54322）。

### 2.4 验证 Supabase 容器运行状态

```bash
docker ps --filter "name=supabase" --format "table {{.Names}}\t{{.Status}}"
```

预期输出 12 个容器，状态均为 `Up` 或 `healthy`：

```
supabase_kong_supabase           Up ... (healthy)
supabase_db_supabase             Up ... (healthy)
supabase_auth_supabase           Up ... (healthy)
supabase_rest_supabase           Up ...
supabase_realtime_supabase       Up ... (healthy)
supabase_storage_supabase        Up ... (healthy)
supabase_studio_supabase         Up ... (healthy)
...
```

## 3. Supabase 后台初始化（数据库建表）

### 3.1 打开 Supabase Studio

浏览器访问：`http://127.0.0.1:54323`

### 3.2 执行建表 SQL

在 Studio 左侧菜单 → **SQL Editor** → 新建查询，依次执行以下 SQL：

#### 3.2.1 用户资料表

```sql
CREATE TABLE public.profiles (
  id UUID REFERENCES auth.users ON DELETE CASCADE PRIMARY KEY,
  nickname TEXT NOT NULL,
  employee_id TEXT,
  avatar_url TEXT,
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now()
);
COMMENT ON TABLE public.profiles IS '用户公开资料表';
COMMENT ON COLUMN public.profiles.id IS '用户 ID，关联 auth.users';
COMMENT ON COLUMN public.profiles.nickname IS '用户昵称（公开展示）';
COMMENT ON COLUMN public.profiles.employee_id IS '7 位工号，仅展示用，非唯一标识';
COMMENT ON COLUMN public.profiles.avatar_url IS '头像图片链接';
```

#### 3.2.2 好友关系表

```sql
CREATE TABLE public.friendships (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  user_id UUID REFERENCES public.profiles(id) ON DELETE CASCADE,
  friend_id UUID REFERENCES public.profiles(id) ON DELETE CASCADE,
  status TEXT DEFAULT 'accepted',
  created_at TIMESTAMPTZ DEFAULT now()
);
COMMENT ON TABLE public.friendships IS '好友关系表';
COMMENT ON COLUMN public.friendships.status IS '状态：pending (申请中) / accepted (已是好友)';
```

#### 3.2.3 聊天消息表

```sql
CREATE TABLE public.messages (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  sender_id UUID NOT NULL REFERENCES public.profiles(id) ON DELETE CASCADE,
  receiver_id UUID NOT NULL REFERENCES public.profiles(id) ON DELETE CASCADE,
  content TEXT NOT NULL,
  msg_type TEXT DEFAULT 'text' CHECK (msg_type IN ('text', 'image', 'file', 'voice')),
  is_read BOOLEAN DEFAULT false,
  created_at TIMESTAMPTZ DEFAULT now()
);
COMMENT ON TABLE public.messages IS '聊天消息表';
COMMENT ON COLUMN public.messages.msg_type IS '消息类型：text | image | file | voice';
COMMENT ON COLUMN public.messages.is_read IS '接收方是否已读';
COMMENT ON COLUMN public.messages.created_at IS '消息发送时间';
```

#### 3.2.4 开启 Realtime 发布

```sql
ALTER PUBLICATION supabase_realtime ADD TABLE public.messages;
```

> 执行后，可在 Studio → Database → **Replication** 中确认 `messages` 表已在发布列表中。

#### 3.2.5 添加 Row Level Security (RLS) 策略

Supabase 默认开启 RLS，必须显式添加策略才能正常读写。执行以下策略（或直接运行 `docs/更新表结构.sql` 第 7 节）：

```sql
-- profiles: 用户可插入自己的资料（注册时写入）
CREATE POLICY "Users can insert own profile"
  ON public.profiles FOR INSERT
  WITH CHECK (auth.uid() = id);

-- profiles: 用户可更新自己的资料
CREATE POLICY "Users can update own profile"
  ON public.profiles FOR UPDATE
  USING (auth.uid() = id)
  WITH CHECK (auth.uid() = id);

-- profiles: 所有人可查看资料（公开信息）
CREATE POLICY "Profiles are viewable by everyone"
  ON public.profiles FOR SELECT
  USING (true);

-- messages: 发送者可插入消息
CREATE POLICY "Users can insert own messages"
  ON public.messages FOR INSERT
  WITH CHECK (auth.uid() = sender_id);

-- messages: 参与对话的双方可查看消息
CREATE POLICY "Users can view their conversations"
  ON public.messages FOR SELECT
  USING (auth.uid() = sender_id OR auth.uid() = receiver_id);

-- friendships: 用户可插入好友关系
CREATE POLICY "Users can insert own friendships"
  ON public.friendships FOR INSERT
  WITH CHECK (auth.uid() = user_id);

-- friendships: 用户可查看自己的好友关系
CREATE POLICY "Users can view own friendships"
  ON public.friendships FOR SELECT
  USING (auth.uid() = user_id OR auth.uid() = friend_id);
```

> 执行后，可在 Studio → Authentication → **Policies** 中确认 7 条策略已生效。

### 3.3 配置 Auth 免邮件确认

在 Studio → Authentication → **Settings** → Email，确认或修改：

```
Enable email confirmations: 关闭
```

本地开发环境下，`supabase start` 已默认开启 `GOTRUE_MAILER_AUTOCONFIRM=true`，注册即激活，无需验证邮件。

## 4. 启动客户端开发服务器

```bash
cd d:\ProjectDir\app-chat\client-chat-tauri
npm run dev
```

Vite 默认在 `http://localhost:5173` 启动，在**浏览器**中调试即可。

> **为什么不是 `npm run tauri dev`？**  
> 当前阶段项目为纯 Vue 3 前端，Tauri 桌面外壳（Rust 源码、.tauri/ 目录、`@tauri-apps/cli`）尚未脚手架。`npm run dev`（即 `vite`）是现阶段唯一的启动方式。所有聊天功能（登录、消息收发、实时推送）在浏览器中完全可调试，无需桌面窗口。待后续集成 Tauri 后，才会新增 `npm run tauri dev` 脚本。

## 6. 常见问题排查

### 6.1 前端启动报错 `缺少环境变量 VITE_SUPABASE_URL`

**原因**：`.env.development` 文件不存在或缺少必要配置。

**解决**：确认 `client-chat-tauri/.env.development` 存在且包含 `VITE_SUPABASE_URL` 和 `VITE_SUPABASE_ANON_KEY`，然后**重启 Vite 开发服务器**（环境变量只在启动时加载）。

### 6.2 注册失败：`Database error...`

**原因**：数据库表未创建。

**解决**：打开 Supabase Studio → SQL Editor，执行第 3 节的建表 SQL。

### 6.3 登录成功但`profiles`查询失败

**原因**：`profiles` 表不存在或已注册用户未写入 `profiles` 记录。

**解决**：
1. 确认 `profiles` 表已创建
2. 在 Studio → Table Editor → profiles 中手动补一条记录（`id` 填 `auth.users` 中用户的 UUID）

### 6.4 实时消息不更新

**原因**：`messages` 表未加入 Realtime 发布。

**解决**：执行 `ALTER PUBLICATION supabase_realtime ADD TABLE public.messages;`

### 6.5 `supabase start` 报容器启动失败

**原因**：Docker Desktop 未运行或端口被占用。

**解决**：
```bash
# 检查 Docker 是否运行
docker info

# 停止全部 Supabase 容器后重试
supabase stop
supabase start
```

### 6.6 Windows 下 Analytics 警告

```
WARNING: Analytics on Windows requires Docker daemon exposed on tcp://localhost:2375
```

**影响**：不影响核心功能（聊天、Auth、Storage），可忽略。如需消除，参考 [Supabase 官方 Windows 指南](https://supabase.com/docs/guides/local-development/cli/getting-started?queryGroups=platform&platform=windows)。

### 6.7 端口冲突

| 端口 | 服务 | 冲突处理 |
|------|------|----------|
| 54321 | Kong API 网关 | 修改 `supabase/config.toml` 中 `api.port` |
| 54322 | PostgreSQL | 修改 `supabase/config.toml` 中 `db.port` |
| 54323 | Studio | 修改 `supabase/config.toml` 中 `studio.port` |
| 5173 | Vite 开发服务器 | 在 `vite.config.ts` 中修改 `server.port` |

## 7. 内网生产部署

### 7.1 Supabase 内网部署

1. 在联网机器上导出镜像：
```bash
docker save -o supabase-images.tar $(docker images --format '{{.Repository}}:{{.Tag}}' | grep supabase)
```

2. 拷贝 `supabase-images.tar` 到内网服务器，导入：
```bash
docker load -i supabase-images.tar
```

3. 修改 `supabase/config.toml` 中的 `JWT_SECRET`、`POSTGRES_PASSWORD` 等敏感配置。

4. 启动：
```bash
supabase start
```

5. 配置内网防火墙允许对应端口（54321、54323 等）。

### 7.2 客户端配置

修改 `client-chat-tauri/.env.production`：

```env
VITE_SUPABASE_URL=http://<内网服务器IP>:54321
VITE_SUPABASE_ANON_KEY=<生产环境 Publishable Key>
```

### 7.3 打包客户端

```bash
cd client-chat-tauri
npm run tauri build
```

产物在 `client-chat-tauri/src-tauri/target/release/bundle/` 目录。
